# Frontend Change Suggestions

这份文档只讨论 `frontend/` 后续应该怎么改，目标是让这套前端更稳定、更容易维护，而不是继续堆补丁。

## 当前判断

`frontend` 现在能用，但已经有几个明显问题：

- 页面逻辑、Docker 运行时配置、下载逻辑、桌面模式逻辑都混在 `app.js` 里。
- 静态资源更新后，容器和浏览器缓存都容易让“代码已经改了，但页面没生效”。
- 下载链路、任务状态链路、Key 保存链路都经历过多次补丁，后续继续叠加会越来越脆。

结论：

- 不建议继续临时修一处补一处。
- 建议把 `frontend` 维持为一套稳定的“面向部署的用户前端”。

## 建议目标

后续前端修改建议优先满足这四点：

- 用户只看到最少必要输入项。
- Docker / 浏览器 / 桌面三种运行方式共用同一套核心逻辑。
- 下载、任务查询、错误展示必须稳定，不依赖浏览器偶发行为。
- 改动后能快速确认“线上到底是不是新版本”。

## 建议一：收敛为单一主前端

现在最应该做的是维持 `frontend/` 作为唯一前端目录，不再重新引入平行目录。

建议：

- 选一套作为主版本，现已统一为 `frontend`。
- 把和 Docker 无关的页面逻辑也统一到同一份 `index.html` / `app.js`。

原因：

- 一旦重新出现两套前端目录，很容易回到“本地正常，Docker 异常”。
- 当前很多问题本质上来自目录分叉而不是业务逻辑本身。

## 建议二：拆分 `app.js`

当前 `frontend/app.js` 太大，职责太杂。

建议至少拆成这几块：

- `config.js`
  负责 `runtime-config`、桌面模式、浏览器本地存储。
- `api.js`
  负责 `fetch`、上传、任务查询、下载。
- `job-view.js`
  负责状态渲染、按钮启用、错误展示、失败诊断展示。
- `app.js`
  只做页面启动和事件绑定。

原因：

- 现在下载、Key 保存、任务状态、桌面桥接都在一个文件里，改一个地方容易带坏别的地方。

## 建议三：下载区改成纯按钮

现在下载区虽然已经尽量改成真正下载，但底层还是基于 `<a>` 元素和 `dataset.url`。

建议：

- 把这四个下载入口全部改成 `<button type="button">`
- 点击时统一走 JS：
  - 查询当前按钮对应资源
  - `fetchProtected`
  - `resp.blob()`
  - `downloadBlob`

建议改造对象：

- `download-btn`
- `pdf-btn`
- `markdown-btn`
- `markdown-raw-btn`

原因：

- `<a>` 天然容易受浏览器默认行为、缓存、target、新窗口策略影响。
- 纯按钮更符合当前“鉴权下载”的实际需求。

## 建议四：给静态资源加版本戳

这是目前非常值得尽快补上的点。

建议：

- `index.html` 中引用静态资源时带版本参数，例如：
  - `app.js?v=20260401`
  - `styles.css?v=20260401`
- Docker 构建时把 git sha 或构建时间写进版本号。

原因：

- 你前面已经反复遇到过：镜像重建了，浏览器还是旧 JS。
- 只靠手工 `Ctrl+F5` 不可靠。

## 建议五：运行时配置只保留最小集合

现在这套用户前端建议只保留：

- `apiBase`
- `xApiKey`
- `mineruToken`
- `modelApiKey`
- `model`
- `baseUrl`

不要再把更多内部参数塞进 `runtime-config.js`。

原因：

- 运行时配置过多会让问题定位很困难。
- 用户版前端应尽量“固定策略 + 少量必填项”。

## 建议六：把“浏览器模式”和“桌面模式”隔离清楚

现在两种模式共用页面是合理的，但状态来源应更清楚。

建议：

- 浏览器模式：
  - Key 存浏览器 `localStorage`
  - API Base 来自 `runtime-config.js`
- 桌面模式：
  - Key 只走 Tauri 后端存储
  - 不依赖浏览器本地存储

建议在代码层明确做两套小函数：

- `loadBrowserConfig() / saveBrowserConfig()`
- `loadDesktopConfig() / saveDesktopConfig()`

不要继续把两种模式揉在同一条判断链里扩展。

## 建议七：失败诊断面板继续增强

这一轮已经补了结构化诊断，但前端展示还可以继续收敛。

建议最终显示固定几项：

- 失败阶段
- 错误类型
- 是否建议重试
- 上游主机
- 建议动作

不要直接把完整 Python traceback 全量暴露给普通用户。

可以增加：

- “查看详细错误” 折叠面板
- 普通视图只展示简洁诊断

## 建议八：增加前端版本自检

建议在页面底部或设置区展示：

- frontend version
- build time
- runtime api base

例如：

```text
frontend build: 2026-04-01 17:20
api base: http://127.0.0.1:40001
```

原因：

- 以后排查“到底是不是新页面”会快很多。

## 建议九：清理目录脏文件

当前前端目录里还有不应该保留的内容，例如：

- `.ipynb_checkpoints/`

建议清理：

- 删除无关缓存文件
- 确保 `.gitignore` 覆盖这些生成物

原因：

- 这类文件会混淆 diff，也容易让你误判当前实际页面版本。

## 建议十：把用户流程固定下来

面向小白用户时，页面流程最好固定：

1. 填两个 key
2. 上传 PDF
3. 运行任务
4. 查状态
5. 下载结果

不要再让页面逐步长回“开发者控制台”。

如果以后确实需要高级参数：

- 单独放一个开发者版前端
- 不要继续往用户版里塞隐藏开关

## 建议优先级

如果只做最值得做的几项，建议顺序如下：

1. 给静态资源加版本戳，解决缓存问题。
2. 下载入口改成纯按钮，不再依赖 `<a>`。
3. 拆分 `app.js`，把配置、API、视图分开。
4. 维持 `frontend/` 单目录结构，避免重新引入平行前端目录。
5. 清理目录脏文件和历史残留。

## 不建议做的事

后续不建议再做这些：

- 继续在用户版前端暴露更多内部参数。
- 继续依赖浏览器默认打开 PDF / JSON 的行为。
- 继续维护两套长期分叉的前端逻辑。
- 把后端内部路径、原始日志、原始 JSON 全量直接暴露给普通用户。

## 一句话结论

如果你后续还要继续维护这套前端，最正确的方向不是继续补丁式修兼容目录，而是把它收敛成：

- 单一主前端
- 固定用户流程
- 纯按钮下载
- 版本可见
- 静态资源可控刷新

这样后面 Docker、桌面版、浏览器版都会轻很多。
