# 产物清单与下载

## 1. 为什么需要产物清单

任务目录是后端内部实现细节，后面可能会继续调整。

所以前端或外部调用方不应该依赖：

- `rendered/` 下面一定有什么
- `md/` 下面一定有什么
- `ocr/unpacked/` 一定不变

现在正式的对外真源是数据库里的稳定产物清单，再通过 API 暴露出来。

## 2. 主接口

主任务：

`GET /api/v1/jobs/{job_id}/artifacts-manifest`

OCR 子任务：

`GET /api/v1/ocr/jobs/{job_id}/artifacts-manifest`

## 3. 它返回什么

它返回这个任务当前已经登记的产物条目，每条条目有：

- `artifact_key`
- `artifact_group`
- `artifact_kind`
- `ready`
- `file_name`
- `content_type`
- `size_bytes`
- `relative_path`
- `checksum`
- `source_stage`
- `updated_at`
- `resource_path`
- `resource_url`

前端真正该用的是：

- `artifact_key`
- `ready`
- `resource_path` / `resource_url`

详情接口里的 `artifacts` 适合快速判断按钮状态；完整机器读取以 `artifacts-manifest` 为准。

## 4. 常见 artifact_key

目前常见包括：

- `source_pdf`
  - 如果请求设置了 `ocr.page_ranges` 且走本地上传，这里指向的是任务内生成的子集 PDF
- `translated_pdf`
- `typst_source`
- `typst_render_pdf`
- `markdown_raw`
- `markdown_images_dir`
- `markdown_bundle_zip`
- `normalized_document_json`
- `normalization_report_json`
- `layout_json`
- `translation_manifest_json`
- `provider_bundle_zip`
- `provider_result_json`
- `pipeline_summary`
- `events_jsonl`
  - 对应任务运行期落盘的事件流导出，适合调试下载

## 5. 推荐读取方式

前端或脚本不要再根据目录结构猜文件位置，而应：

1. 先请求 `artifacts-manifest`
2. 找到目标 `artifact_key`
3. 检查 `ready`
4. 使用 `resource_path` 或 `resource_url`

其中 `markdown_bundle_zip` 是最适合前端直接下载的 Markdown 包：

- 只包含 `markdown/full.md`
- 只包含 `markdown/images/**`
- 不包含原始 PDF
- 不包含翻译后 PDF
- 不包含 provider bundle
- 不包含其他调试 JSON

如果你希望 ZIP 解压后目录带任务号，可以在直连下载时加：

`?include_job_dir=true`

此时根目录会从默认的 `markdown/` 变成：

`{job_id}-markdown/`

如果你想下载任务的原始事件流，也可以直接查：

- `artifact_key = events_jsonl`

它的语义是：

- 一个稳定调试产物
- 内容来自任务日志目录下的 `events.jsonl`
- 适合排错、导出、归档

但注意：

- 前端主界面的“事件流”tab 仍然应该优先读取 `/events` 接口
- 不建议前端把 `events_jsonl` 当成主展示接口真源

## 6. 和旧接口的关系

旧接口还保留，比如：

- `/pdf`
- `/markdown`
- `/normalized-document`

但内部已经开始向“先查产物清单，再定位真实文件”收敛。

可以把它理解成：

- 旧接口是稳定下载入口
- `artifacts-manifest` 是稳定发现入口
- 详情里的 `artifacts.pdf` / `artifacts.markdown` / `artifacts.bundle` 是推荐的页面状态字段
- 详情里的同级 `pdf_url` / `markdown_url` / `bundle_url` 等旧字段只是兼容别名，不建议新代码继续依赖

但要注意一个边界：

- 这里的“旧接口”指的是当前 API 里的稳定资源入口
- 不是指旧任务目录布局兼容

如果任务本身仍来自 `originPDF/jsonPDF/transPDF/typstPDF` 旧布局，或者 artifact 还存绝对路径，那么这些下载入口也会直接拒绝，必须重跑

## 7. 为什么这样做更稳

因为以后即使后端把文件从：

- `rendered/...`

改到：

- `outputs/render/...`

只要后端更新数据库登记逻辑，外部接口就不需要跟着改。

这正是这次重构的目的：把“目录结构”从公开契约里剥离出去。
