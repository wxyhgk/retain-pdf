# 产物清单与下载

## 1. 为什么需要产物清单

任务目录是后端内部实现细节，后面可能会继续调整。

所以前端或外部调用方不应该依赖：

- `rendered/` 下面一定有什么
- `md/` 下面一定有什么
- `ocr/unpacked/` 一定不变

现在正式的对外真源是数据库里的稳定产物清单，再通过 API 暴露出来。

## 2. 主接口

主任务：

`GET /api/v1/jobs/{job_id}/artifacts-manifest`

OCR 子任务：

`GET /api/v1/ocr/jobs/{job_id}/artifacts-manifest`

## 3. 它返回什么

它返回这个任务当前已经登记的产物条目，每条条目有：

- `artifact_key`
- `artifact_group`
- `artifact_kind`
- `ready`
- `file_name`
- `content_type`
- `size_bytes`
- `relative_path`
- `source_stage`
- `resource_path`
- `resource_url`

前端真正该用的是：

- `artifact_key`
- `ready`
- `resource_path` / `resource_url`

## 4. 常见 artifact_key

目前常见包括：

- `source_pdf`
  - 如果请求设置了 `ocr.page_ranges` 且走本地上传，这里指向的是任务内生成的子集 PDF
- `translated_pdf`
- `typst_source`
- `typst_render_pdf`
- `markdown_raw`
- `markdown_images_dir`
- `markdown_bundle_zip`
- `normalized_document_json`
- `normalization_report_json`
- `layout_json`
- `provider_bundle_zip`
- `provider_result_json`
- `pipeline_summary`

## 5. 推荐读取方式

前端或脚本不要再根据目录结构猜文件位置，而应：

1. 先请求 `artifacts-manifest`
2. 找到目标 `artifact_key`
3. 检查 `ready`
4. 使用 `resource_path` 或 `resource_url`

其中 `markdown_bundle_zip` 是最适合前端直接下载的 Markdown 包：

- 只包含 `markdown/full.md`
- 只包含 `markdown/images/**`
- 不包含原始 PDF
- 不包含翻译后 PDF
- 不包含 provider bundle
- 不包含其他调试 JSON

如果你希望 ZIP 解压后目录带任务号，可以在直连下载时加：

`?include_job_dir=true`

此时根目录会从默认的 `markdown/` 变成：

`{job_id}-markdown/`

## 6. 和旧接口的关系

旧接口还保留，比如：

- `/pdf`
- `/markdown`
- `/normalized-document`

但内部已经开始向“先查产物清单，再定位真实文件”收敛。

可以把它理解成：

- 旧接口是稳定下载入口
- `artifacts-manifest` 是稳定发现入口

## 7. 为什么这样做更稳

因为以后即使后端把文件从：

- `rendered/...`

改到：

- `outputs/render/...`

只要后端更新数据库登记逻辑，外部接口就不需要跟着改。

这正是这次重构的目的：把“目录结构”从公开契约里剥离出去。
