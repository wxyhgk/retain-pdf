# 协同开发约定

这份文档不是接口文档，而是给后端协同开发用的边界约定。

目标只有一个：

- 多人同时开发 `rust_api` 时，尽量做到各改各的，不互相踩职责，不把结构再写乱

## 1. 模块职责

当前默认按下面的边界协作：

- `routes/*`
  - 只做 HTTP adapter
  - 负责请求解析、鉴权后入口、响应包装
  - 不负责业务聚合，不直接拼接 job command
- `services/jobs/*`
  - 负责任务域内业务逻辑
  - 包括 query、presentation、creation、control
  - 这里是大多数“业务改动”优先落点
- `services/job_factory.rs`
  - 负责 job snapshot 装配与启动边界
  - 默认分成“build snapshot”和“start execution”两步
- `job_runner/*`
  - 负责运行态执行
  - 包括排队、取消、进程拉起、OCR 子任务衔接、render/translate 执行链
- `models/*`
  - 只放 DTO、输入输出模型、持久化模型
  - 不新增业务编排或文件系统读取逻辑

## 2. 依赖规则

默认依赖方向：

- `routes -> services -> job_factory / job_runner / db`
- `job_runner -> db / config / runtime state`
- `models` 不反向依赖 `routes` 或 `services`

默认禁止：

- `routes` 直接写复杂业务逻辑
- `models/view.rs` 重新长出文件读取、失败诊断聚合之类逻辑
- 纯装配 helper 继续直接吃 `AppState`

## 3. AppState 使用约定

`AppState` 允许出现的地方：

- 路由入口
- 生命周期入口
- 真正需要协调取消、执行槽位、进程拉起的运行态模块

优先改成窄依赖的地方：

- 命令构建
- snapshot 装配
- 上传校验与存储
- 只读 view 聚合
- 只做落库/落事件的 helper

如果一个函数只需要下面这些资源，就不要传 `AppState`：

- `&Db`
- `&AppConfig`
- `&Path`
- `&RwLock<HashSet<String>>`
- `&Arc<Semaphore>`

## 4. 新需求落点规则

多人协作时，先判断需求属于哪类，再落代码：

- 新增查询字段、详情展示、产物视图
  - 优先改 `services/jobs/presentation` 或 `services/jobs/query`
- 新增创建参数、提交流程、bundle 逻辑
  - 优先改 `services/jobs/creation`
- 新增执行阶段、子任务衔接、取消语义
  - 优先改 `job_runner/*`
- 新增公共响应字段或输入字段
  - 改 `models/*`
- 只有 HTTP 入参/出参变化
  - 改 `routes/*`，但尽量不要把业务判断留在 route

## 5. 常见反模式

下面这些属于协作中最容易把结构写坏的方式：

- 在 route 里顺手读 DB、拼 view、拼 command、做文件系统判断
- 在 `services` 里直接写 HTTP 语义
  - 例如直接操心 Header、Multipart、Response
- 在 `job_runner` 子模块里新增新的 `AppState` 透传 helper
- 在 `models` 里新增“顺手把磁盘文件读了”的聚合逻辑
- 为了省事把多个职责重新塞回一个黑盒函数

## 6. 提交前最低检查

如果改到了 `rust_api`，默认至少做下面几项里的相关项：

- `cargo check --quiet`
- 改了 creation / job_factory：补对应 service 测试
- 改了 process runner / lifecycle：补对应 runner 测试
- 改了 query / presentation：补对应 query 或 presentation 测试

如果一个改动让下层 helper 又开始直接依赖 `AppState`，默认应在评审里解释为什么必须这样做。
