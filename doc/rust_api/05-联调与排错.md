# 联调与排错

## 1. 最常见的三类误判

### 1.1 后端没返回，还是前端没读对

先直接用 curl 看原始响应，不要先猜。

例如：

```bash
curl -s http://127.0.0.1:41000/api/v1/jobs/<job_id> \
  -H 'X-API-Key: your-key'
```

以及：

```bash
curl -s 'http://127.0.0.1:41000/api/v1/jobs/<job_id>/events?limit=50&offset=0' \
  -H 'X-API-Key: your-key'
```

如果原始响应有值，而页面没有，优先查前端解包路径。

### 1.2 “后端未返回 runtime.stage_history”

这句话有两种完全不同的情况：

第一种，当前任务是历史老任务：

- `runtime = null`
- 说明当时后端还未持久化运行态时间线

第二种，当前任务是新任务但后端真的没写：

- 这才是需要排查的异常

不要把这两种情况混成一个问题。

### 1.3 “事件流为空”

优先检查：

1. 接口是不是 `200`
2. `data.items` 是否非空
3. 前端是不是错误读取为顶层 `items`

## 2. 任务详情页的推荐取值表

- 当前状态：`data.status`
- 当前阶段：`data.runtime.current_stage`，兜底 `data.stage`
- 当前阶段说明：`data.stage_detail`
- 过程时间线：`data.runtime.stage_history`
- 当前阶段耗时：`data.runtime.active_stage_elapsed_ms`
- 总耗时：`data.runtime.total_elapsed_ms`
- 失败摘要：`data.failure.summary`
- 失败类别：`data.failure.category`
- 下载按钮：`data.actions.*.enabled`

## 3. 事件流 tab 的推荐取值表

- 事件数组：`data.items`
- 分页 limit：`data.limit`
- 分页 offset：`data.offset`

每个 item 重点展示：

- `seq`
- `ts`
- `level`
- `stage`
- `event`
- `message`

`payload` 建议作为展开区，不要默认全部摊开。

## 4. 磁盘上怎么查

任务根目录一般在：

`DATA_ROOT/jobs/{job_id}/`

常见排查位置：

- `logs/events.jsonl`
- `artifacts/pipeline_summary.json`
- `ocr/`
- `translated/`
- `rendered/`

## 5. 一个真实案例

任务 `20260404150516-75857c`：

- 详情接口里 `runtime = null`
- 所以“过程时间线”不能显示
- 但 `/events` 接口是有数据的
- 磁盘上也有 `logs/events.jsonl`

这说明：

- 该任务属于运行态时间线持久化上线前的历史任务
- 不是事件流坏了
- 也不是前端时间线 tab 自己渲染失败

## 6. 文档级结论

前端如果只需要稳定展示：

- 时间线只看 `runtime.stage_history`
- 事件流只看 `/events`
- 不要互相回填，不要混用，不要从事件流反推主时间线

## 7. OCR 页码范围联调要点

`POST /api/v1/jobs` 里如果要限制 OCR 页码范围，字段位置是：

```json
{
  "ocr": {
    "page_ranges": "1-5"
  }
}
```

当前后端行为：

- `ocr.page_ranges` 会进入任务持久化的 `request_payload`
- `provider=mineru` 时：
  - `source.upload_id` 上传 PDF 路径会先裁剪出子集 PDF，再上传该子集
  - `source.source_url` 远程 URL 路径也会透传
- 空字符串表示不限制页码

如果 `source.upload_id + ocr.page_ranges` 已经生效，那么任务目录里的 `source_pdf`、后续翻译页码和最终渲染 PDF 都应该只覆盖这个子集，不应该再出现“整本 PDF 里只有部分页被翻译”的结果。

如果怀疑页码范围没生效，先查这三层，不要直接猜前端：

1. 请求详情里 `request_payload.ocr.page_ranges` 是否非空
2. 数据库 `jobs.request_json` 中 `$.ocr.page_ranges` 是否非空
3. 再看 provider 执行日志，而不是先怀疑页面表单
