# 任务列表接口

## 1. 主接口

主任务列表：

`GET /api/v1/jobs`

OCR 子任务列表：

`GET /api/v1/ocr/jobs`

这两个接口都适合做：

- 首页“最近任务”
- 任务历史列表
- 简单筛选后的任务面板

## 2. 查询参数

当前支持：

- `limit`
- `offset`
- `status`
- `workflow`
- `provider`

说明：

- `limit`
  - 可选，默认使用后端内置默认值
- `offset`
  - 可选，默认 `0`
- `status`
  - 可选
  - 当前值：`queued` / `running` / `succeeded` / `failed` / `canceled`
- `workflow`
  - 可选
  - 当前值：`mineru` / `ocr`
- `provider`
  - 可选
  - 当前主要用于按 OCR provider 诊断信息过滤，例如 `mineru`

## 3. 排序规则

当前固定规则：

- 按 `updated_at DESC`

也就是说：

- 最近有变化的任务排最前
- 更适合“最近任务”面板
- 不等于“最近创建”

## 4. 返回结构

返回包装：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "items": [
      {
        "job_id": "20260406063244-2176e4",
        "display_name": "paper.pdf",
        "workflow": "mineru",
        "status": "running",
        "trace_id": "trace-abc",
        "stage": "translating",
        "invocation": {
          "stage": "mineru",
          "input_protocol": "stage_spec",
          "stage_spec_schema_version": "mineru.stage.v1"
        },
        "created_at": "2026-04-06T06:32:44Z",
        "updated_at": "2026-04-06T06:33:00Z",
        "detail_path": "/api/v1/jobs/20260406063244-2176e4",
        "detail_url": "http://127.0.0.1:41000/api/v1/jobs/20260406063244-2176e4"
      }
    ],
    "invocation_summary": {
      "stage_spec_count": 1,
      "unknown_count": 0
    }
  }
}
```

每个 item 固定包含：

- `job_id`
- `display_name`
- `workflow`
- `status`
- `trace_id`
- `stage`
- `created_at`
- `updated_at`
- `detail_path`
- `detail_url`
- `invocation`

列表响应还会额外给一个聚合字段：

- `data.invocation_summary`

## 5. 字段怎么理解

- `job_id`
  - 任务唯一标识
- `display_name`
  - 面向用户展示的任务名称
  - 取值优先级：
    1. 上传源 PDF 文件名
    2. 远程 URL 最后文件名
    3. 最后兜底 `job_id`
- `workflow`
  - 主任务通常是 `mineru`
  - OCR 子任务通常是 `ocr`
- `status`
  - 当前状态或终态
- `trace_id`
  - 当前任务挂在 artifacts 上的 trace id，可能为空
- `stage`
  - 当前粗粒度阶段
  - 适合做列表标签，不适合代替详情里的 `stage_detail`
- `detail_url`
  - 前端点开详情后继续请求的主入口
- `invocation`
  - 列表级执行协议摘要
  - 新任务应显示 `input_protocol=stage_spec`
  - 适合做调试标签，不建议替代详情里的完整判断
- `invocation_summary`
  - 当前分页结果里的协议聚合统计
  - 适合前端做“这一页是不是都已经切到新协议”这类轻量提示

## 6. 前端推荐读取方式

最近任务列表：

`GET /api/v1/jobs?limit=20&offset=0`

只看失败任务：

`GET /api/v1/jobs?status=failed&limit=20&offset=0`

只看 OCR 子任务：

`GET /api/v1/ocr/jobs?limit=20&offset=0`

推荐展示：

- 标题：`display_name`
- 状态：`status`
- 当前阶段：`stage`
- 协议标签：`invocation.input_protocol`
- 最近更新时间：`updated_at`
- 点击后进入：`detail_url`

## 7. 不要指望列表接口返回什么

当前列表接口不直接返回：

- `stage_detail`
- `runtime`
- `runtime.stage_history`
- `failure.summary`
- `artifacts`

所以如果前端想展示：

- 失败摘要
- 总耗时
- 下载按钮
- 过程时间线

应当在点击列表项后，继续请求详情接口，而不是要求列表接口承载所有信息。

## 8. OCR 列表和主列表的关系

`GET /api/v1/ocr/jobs` 本质上只是：

- 复用同一套列表逻辑
- 额外固定 `workflow=ocr`

所以：

- 返回结构一致
- 只是数据范围不同

## 9. 为什么这样设计

这样做的目的很直接：

- 列表接口保持轻量
- 详情接口承载完整状态
- 前端首页先快，再在详情页拉全量数据

这比一上来就把 `runtime`、`failure`、`artifacts` 全堆进列表更稳，也更容易长期维护。
