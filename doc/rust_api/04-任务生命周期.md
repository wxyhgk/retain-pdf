# 任务生命周期

## 1. 当前支持的 workflow

Rust API 现在支持 4 类 workflow：

1. `mineru`
   完整链路：OCR -> Normalize -> Translate -> Render
2. `ocr`
   只跑 OCR/Normalize，产出 `document.v1.json`
3. `translate`
   OCR -> Normalize -> Translate，停在翻译产物，不进入渲染
4. `render`
   复用已有 job artifacts，只重跑渲染

## 2. 主链路

### 2.1 `mineru`

1. 上传 PDF
2. 创建主任务
3. 主任务内部创建 OCR 子任务
4. OCR provider 上传、轮询、下载结果
5. 标准化为 `document.v1`
6. 翻译
7. 渲染
8. 输出 PDF / ZIP / 其他产物

### 2.2 `translate`

1. 上传 PDF
2. 创建翻译任务
3. 翻译任务内部创建 OCR 子任务
4. OCR provider 上传、轮询、下载结果
5. 标准化为 `document.v1`
6. 翻译
7. 输出 translation payload、`translation-manifest.json`、诊断信息

### 2.3 `render`

1. 创建渲染任务
2. 请求里通过 `source.artifact_job_id` 指向一个已有 job
3. 后端从该 job 复用 `source_pdf` 与 `translations_dir`
4. 只执行渲染
5. 输出新的 PDF / Typst / 渲染产物

## 3. 为什么会有 OCR 子任务

`mineru` 和 `translate` 主任务都不直接把所有事情堆在一起做，而是会产生一个 OCR 子任务，通常是：

`{job_id}-ocr`

这样做有两个好处：

- OCR transport 和主翻译渲染链路可以分开观察
- OCR 的 provider 诊断可以单独挂在详情里返回

## 4. 常见阶段名

阶段名会随着链路推进变化，常见包括：

- `queued`
- `ocr_submitting`
- `ocr_upload`
- `mineru_upload`
- `mineru_processing`
- `translation_prepare`
- `normalizing`
- `translating`
- `domain_inference`
- `page_policies`
- `rendering`
- `finished`
- `failed`

补充说明：

- `translate` 一般会停在 `translating -> finished`
- `render` 一般会直接进入 `rendering`
- `ocr` 不会进入 `translating` 或 `rendering`

不是每条任务都会经过完全相同的阶段，但主思路一致。

## 5. 请求侧怎么区分

最关键的是两件事：

1. `workflow`
   决定这是完整任务、OCR-only、Translate-only 还是 Render-only
2. `source`
   - `upload_id`：用于 `mineru` / `translate`
   - `artifact_job_id`：用于 `render`

当前约定：

- `workflow=translate` 仍然会创建 OCR 子任务，但不会进入渲染
- `workflow=render` 不会再跑 OCR 或翻译，而是复用已有 job artifacts
- `workflow=ocr` 使用 `/api/v1/ocr/jobs` 独立入口，不混在 `/api/v1/jobs` 里提交
- `/api/v1/jobs` 的 JSON 请求体以 `source / ocr / translation / render / runtime` 分组结构为正式契约
- 历史扁平字段只保留在少数 multipart 辅助入口里，不再作为主 JSON 契约宣传

## 6. 前端怎么理解阶段

建议区分三层：

- `status`：任务是否结束
- `stage`：当前在哪个阶段
- `stage_detail`：当前阶段的人类说明

例如：

- `status = running`
- `stage = translating`
- `stage_detail = 已完成第 18/55 批翻译`

这说明：

- 任务还没结束
- 当前阶段是翻译
- 当前批次进度已经走到 18/55

## 7. 下载按钮什么时候可点

前端不要自己拼规则，而要直接看详情接口里的：

- `actions.*.enabled`
- `artifacts.*.ready`

例如：

- PDF 下载按钮看 `actions.download_pdf.enabled`
- ZIP 下载按钮看 `actions.download_bundle.enabled`

补充：

- `translate` 任务成功后，通常只有翻译目录和 bundle ready，PDF 不一定 ready
- `render` 任务成功后，PDF 应该 ready，但不一定有 OCR 相关下载物

## 8. 失败时应该看什么

失败任务优先看：

- `failure.category`
- `failure.summary`
- `failure.root_cause`
- `failure.retryable`
- `failure.suggestion`

如果还不够，再看：

- `runtime.final_failure_category`
- `runtime.final_failure_summary`
- `log_tail`
- `/events`

## 9. 时间线和事件流的职责分工

这个分工非常重要：

- `runtime.stage_history` 回答“全过程每个阶段花了多久”
- `/events` 回答“过程中发生过哪些具体事件”
- `failure` 回答“这次失败被归类成什么、是否可重试、建议怎么处理”
- `failure_diagnostic` 只是兼容旧前端的简化视图

前者适合概览，后者适合排错。
