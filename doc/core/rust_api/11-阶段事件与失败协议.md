# 阶段事件与失败协议

这份文档描述 `Rust API` 与 `Python worker` 当前已经采用的正式事件/失败口径，以及仍然保留的兼容字段。

目标不是再发明一套新协议，而是明确：

- 哪些字段是正式源
- 哪些字段只是兼容投影
- 主链路阶段和 provider 私有阶段怎么分层

## 1. 当前正式口径

截至 2026-04-25，这套协议已经按下面方式落地：

- Python worker 会写 `logs/pipeline_events.jsonl`
- Rust 查询层会合并 DB events 和 `pipeline_events.jsonl`
- Rust detail/list 会优先使用 `pipeline_events.jsonl` 里的 live stage 快照
- Rust / Python 都已支持结构化 `failure`
- `failure_diagnostic` 仍保留，但只作为兼容投影

也就是说，当前正式源已经不是：

- 旧的 `events` 日志文件口径
- 纯 stdout 文本猜阶段
- 只靠 `failure_diagnostic` 表达失败

## 2. 阶段分层规则

正式顶层 `stage` 只表示统一主链路阶段：

- `queued`
- `startup`
- `ocr_submitting`
- `ocr_processing`
- `normalization`
- `translation_prepare`
- `domain_inference`
- `continuation_review`
- `page_policies`
- `translating`
- `render_prepare`
- `rendering`
- `saving`
- `finished`
- `failed`
- `canceled`

provider 私有状态只能出现在：

- `provider`
- `provider_stage`

例如：

```json
{
  "stage": "ocr_processing",
  "provider": "mineru",
  "provider_stage": "mineru_processing"
}
```

因此：

- `mineru_upload`、`mineru_processing`、`paddle_running`

这些都不能再作为正式顶层 `stage` 示例使用。

## 3. 正式事件对象

当前事件查询接口会返回统一事件对象；老字段保留，新字段优先。

示例：

```json
{
  "job_id": "job-123",
  "seq": 12,
  "ts": "2026-04-24T10:12:33Z",
  "level": "info",
  "stage": "translating",
  "stage_detail": "已完成第 18/55 批翻译",
  "provider": "paddle",
  "provider_stage": "",
  "event_type": "stage_progress",
  "event": "stage_progress",
  "message": "已完成第 18/55 批翻译",
  "progress_current": 18,
  "progress_total": 55,
  "retry_count": 0,
  "elapsed_ms": 193822,
  "payload": {
    "origin": "python"
  }
}
```

字段含义：

- `stage`
  主链路统一阶段
- `stage_detail`
  给前端直接显示的阶段细节
- `provider`
  当前 provider 名称
- `provider_stage`
  provider 私有状态
- `event_type`
  正式事件类型
- `event`
  legacy 兼容字段；默认与 `event_type` 同步
- `message`
  legacy 兼容文案；正式语义优先看 `stage_detail` 和 `event_type`
- `progress_current` / `progress_total`
  正式进度字段
- `payload`
  扩展字段容器，不承载主语义

当前推荐的 `event_type`：

- `job_created`
- `status_changed`
- `stage_transition`
- `stage_progress`
- `artifact_published`
- `job_error`
- `job_terminal`
- `diagnostic`

## 4. 正式失败对象

当前 job detail 返回中的正式失败对象是 `failure`。

示例：

```json
{
  "failed_stage": "translation_prepare",
  "provider": "mineru",
  "provider_stage": "mineru_processing",
  "failure_code": "auth_failed",
  "failure_category": "auth",
  "provider_code": "A0211",
  "summary": "鉴权失败",
  "root_cause": "token expired",
  "retryable": false,
  "upstream_host": "mineru.example.test",
  "suggestion": "检查 provider token",
  "last_log_line": "token expired during mineru_processing",
  "raw_excerpt": "token expired"
}
```

字段分层：

- `failed_stage`
  顶层统一失败阶段
- `provider` / `provider_stage`
  provider 侧归属信息
- `failure_code`
  稳定机器码
- `failure_category`
  粗粒度错误分组
- `provider_code`
  上游 provider 原始错误码
- `summary` / `root_cause` / `suggestion`
  面向前端和操作者的可读信息
- `raw_excerpt`
  脱敏后的原始片段

## 5. 兼容字段策略

当前仍然保留这些字段：

- `JobDetailView.stage`
- `JobDetailView.stage_detail`
- `JobDetailView.failure_diagnostic`
- `JobEventRecord.event`
- `JobEventRecord.message`
- `JobEventRecord.payload`

但它们的角色已经固定：

- `stage` / `stage_detail`
  仍是对外主展示字段，但数据源优先来自 live pipeline events
- `failure_diagnostic`
  只作为 `failure` 的 legacy projection
- `event`
  兼容旧客户端
- `message`
  兼容旧客户端或调试展示
- `payload`
  扩展信息，不承担正式主语义

推荐映射关系：

- `failure_diagnostic.failed_stage` -> `failure.failed_stage`
- `failure_diagnostic.error_kind` -> `failure.failure_code`
- `failure_diagnostic.summary` -> `failure.summary`
- `failure_diagnostic.root_cause` -> `failure.root_cause`
- `failure_diagnostic.retryable` -> `failure.retryable`

## 6. Rust 当前消费方式

Rust 现在已经有两层正式消费：

1. event list
   合并 DB events 和 `pipeline_events.jsonl`
2. live stage
   detail/list 从统一事件流中选取最新可展示阶段快照

当前优先级规则：

- 若 `pipeline_events.jsonl` 中存在可用 live stage，则优先于 DB 中过期的 `job.stage`
- `artifact_published` 不会覆盖 live stage 展示
- failure 相关输出优先取正式 `failure` 字段，不再反推旧 `failure_diagnostic`

## 7. 剩余未完成项

还没有彻底做完的只剩这几类：

- 更细粒度的 provider 轮询事件还没有全部标准化成统一 `event_type`
- stdout parser 仍然存在，但角色已经收缩为兼容补充，而不是正式事件源
- 少数历史文档和说明可能还残留旧词，需要继续清理

## 8. 验收标准

如果下面几点都成立，就说明协议已经稳定：

- 前端不需要再从 `log_tail` 猜失败原因
- detail / list / events 对同一任务使用同一套 live stage 优先级
- provider 私有阶段只出现在 `provider_stage`
- `failure_diagnostic` 只作为兼容字段存在
- 新 provider 接入时，只需要填充统一 `stage` / `event_type` / `failure` 协议，而不是再定义一套新口径
