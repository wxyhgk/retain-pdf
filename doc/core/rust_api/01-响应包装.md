# Rust API 响应包装

## 1. 顶层格式

Rust API 当前统一使用同一套响应包装。

成功响应：

```json
{
  "code": 0,
  "message": "ok",
  "data": {}
}
```

失败响应：

```json
{
  "code": 400,
  "message": "具体错误信息"
}
```

读取规则很简单：

- `code = 0` 表示成功
- `message` 是给人看的
- 业务数据永远在 `data`

## 2. 最容易犯错的点

很多前端空白页、空列表、字段“明明后端有但前端没显示”，最后都不是接口没返回，而是读取路径错了。

典型错误：

```json
{
  "items": [...]
}
```

这不是当前 Rust API 的结构。

真正的结构是：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "items": [...]
  }
}
```

也就是说：

- 不能读 `response.items`
- 也不能读 `response.data.items`，如果你的 HTTP 客户端最外层还有一层对象
- 最终应读到 API 返回体里的 `data`

## 3. 事件流接口的具体例子

`GET /api/v1/jobs/{job_id}/events?limit=50&offset=0`

实际返回结构是：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "items": [
      {
        "job_id": "20260404150516-75857c",
        "seq": 1,
        "ts": "2026-04-04T15:05:16Z",
        "level": "info",
        "stage": "queued",
        "event": "job_created",
        "message": "任务已创建",
        "payload": {
          "stage": "queued",
          "status": "queued",
          "workflow": "book"
        }
      }
    ],
    "limit": 50,
    "offset": 0
  }
}
```

这里的关键是：

- 事件数组在 `data.items`
- `limit` 在 `data.limit`
- `offset` 在 `data.offset`

## 4. 建议的前端解包顺序

无论是任务详情、列表还是事件流，都建议按这个顺序处理：

1. 先检查 HTTP 状态码
2. 再检查响应体里的 `code`
3. `code === 0` 时，再读取 `data`
4. 永远不要直接假设业务字段在顶层

## 5. 适合直接写进前端的判断逻辑

- 如果 HTTP 不是 `2xx`，按网络或服务错误处理
- 如果 HTTP 是 `2xx` 但 `code !== 0`，按业务错误处理
- 如果 `code === 0` 且 `data == null`，按“成功但无业务载荷”处理
- 如果字段应在 `data.xxx`，就不要去顶层找
