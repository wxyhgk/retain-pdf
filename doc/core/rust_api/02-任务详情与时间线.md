# 任务详情与时间线

## 1. 主接口

任务详情页的主接口是：

`GET /api/v1/jobs/{job_id}`

详情页的大部分核心信息都应从这一个接口拿：

- 当前状态
- 当前阶段
- 当前阶段说明
- 运行态时间线
- 失败归因
- 下载按钮状态
- OCR 子任务摘要
- 当前任务的执行协议摘要
- 归一化摘要与术语表摘要

## 2. 前端最该关心的字段

重点字段在 `data` 里：

- `request_payload`
- `status`
- `stage`
- `stage_detail`
- `runtime`
- `failure`
- `actions`
- `artifacts`
- `timestamps`
- `log_tail`
- `normalization_summary`
- `glossary_summary`
- `invocation`

其中：

- `request_payload` 是后端为该任务实际保存的请求参数快照
- 联调时如果要确认某个参数到底有没有进后端，优先看这里
- 例如 OCR 页码范围应读：`data.request_payload.ocr.page_ranges`
- 公式翻译模式应读：`data.request_payload.translation.math_mode`
- `invocation` 用来回答“这个任务到底是旧路径还是当前 stage spec 路径跑出来的”
- `normalization_summary` 用来回答“归一化结果是否是当前 `schema_version=1.1`、是否发生了默认值收口”
- `glossary_summary` 用来回答“术语表是否启用、命中与未命中情况如何”

## 3. 新任务的执行协议怎么看

当前新版任务应当看到：

```json
{
  "invocation": {
    "stage": "provider",
    "input_protocol": "stage_spec",
    "stage_spec_schema_version": "provider.stage.v1"
  }
}
```

联调时的判断原则：

- `input_protocol=stage_spec` 说明该任务走的是当前 spec-only worker
- `stage_spec_schema_version` 说明具体阶段 spec 版本
- 如果这里拿不到，不一定是接口坏了，也可能是历史任务没有留下这部分摘要

## 4. 时间线真源

“概览 -> 过程时间线”必须以 `runtime.stage_history` 为准。

不要用 `/events` 反推主时间线。原因很直接：

- `runtime.stage_history` 已经是后端整理好的阶段段落
- 每段都有进出时间
- 每段可以直接挂终态信息
- 前端不需要再做合并、去重、推断

## 5. `runtime` 里有哪些关键字段

当前重点字段：

- `current_stage`
- `stage_started_at`
- `last_stage_transition_at`
- `total_elapsed_ms`
- `active_stage_elapsed_ms`
- `retry_count`
- `last_retry_at`
- `terminal_reason`
- `final_failure_category`
- `final_failure_summary`
- `stage_history`

## 6. `stage_history` 的固定结构

每一项都是一个阶段时间片段：

```json
{
  "stage": "translating",
  "detail": "正在翻译，第 12/22 批",
  "enter_at": "2026-04-04T15:31:02Z",
  "exit_at": null,
  "duration_ms": null,
  "terminal_status": null
}
```

字段含义：

- `stage`：阶段名
- `detail`：进入该阶段时的主要说明
- `enter_at`：进入该阶段的时间
- `exit_at`：离开该阶段的时间；当前活跃阶段通常为空
- `duration_ms`：阶段已完成时才稳定；当前活跃阶段通常为空
- `terminal_status`：若该阶段就是结束前最后阶段，可标记 `succeeded / failed / canceled`

## 7. 运行中任务怎么读

如果任务还在运行：

- 当前活跃阶段也会出现在 `stage_history`
- 该项通常 `exit_at = null`
- 该项通常 `duration_ms = null`
- 当前阶段实时耗时应读 `runtime.active_stage_elapsed_ms`

也就是说，前端不要自己用“当前时间 - enter_at”硬算展示值，除非只是本地补帧；接口真源仍应是 `active_stage_elapsed_ms`。

## 8. 已结束任务怎么读

如果任务已结束：

- `status` 会是 `succeeded / failed / canceled`
- `runtime.terminal_reason` 会说明终态原因
- `runtime.total_elapsed_ms` 是整条链路总耗时
- `runtime.stage_history` 最后一项通常会带 `terminal_status`

## 9. `normalization_summary` 怎么理解

当前任务详情里的 `normalization_summary` 是轻量摘要，不是完整报告。

重点字段：

- `provider`
- `detected_provider`
- `schema`
- `schema_version`
- `document_defaults`
- `page_defaults`
- `block_defaults`
- `page_count`
- `block_count`

推荐前端用法：

- 页面展示只读这个摘要
- 真要排查 adapter/defaults/validation 细节，再下载 `artifacts.normalization_report`

注意：

- 当前主链只接受 `schema_version=1.1`
- 这里的 `*_defaults` 表示默认值收口数量，不再叫 `compat_*`

## 10. 历史任务为什么可能没有时间线

这是最近最容易误判的地方。

历史老任务可能出现：

```json
{
  "runtime": null
}
```

这不表示：

- 前端读错了
- 当前后端坏了
- 当前任务写入失败了

它只表示：

- 这条任务创建和执行时，后端还没有把运行态时间线持久化到数据库

所以这类任务前端应视为“历史数据缺失”。

## 11. “历史任务缺字段”和“旧任务被拒绝”不是一回事

这是最近联调时最容易混淆的两个概念：

- `runtime = null`
  - 这是历史任务数据缺失
  - 任务本身仍可能能看详情、看事件流
- 详情/下载接口直接提示旧任务不支持
  - 这说明该任务还停留在旧目录布局或旧 artifact 存储方式
  - 当前后端不会再帮它做兼容迁移，只能重跑

## 12. 保证范围

当前文档里的保证范围是：

- 任务由新版后端创建
- 任务由同一套新版后端全程执行

只有在这个范围内，`runtime.stage_history` 才保证是完整全过程，而不是只保留最后阶段。

## 13. 适合前端直接使用的读取顺序

1. 先读 `status`
2. 再读 `runtime.current_stage`
3. 时间线直接读 `runtime.stage_history`
4. 当前阶段耗时读 `runtime.active_stage_elapsed_ms`
5. 总耗时读 `runtime.total_elapsed_ms`
6. 失败任务优先读 `failure.summary` 和 `failure.category`
7. 执行协议读 `invocation`
8. 归一化摘要读 `normalization_summary`
9. 术语表摘要读 `glossary_summary`

## 14. `translation.math_mode` 怎么理解

当前翻译链路支持两个值：

- `placeholder`
  - 默认值
  - 继续走旧的公式保护 / placeholder / 恢复链路
- `direct_typst`
  - 实验模式
  - 不做公式 placeholder 保护，直接让模型输出正文和 `$...$` 数学

前端只需要按字符串展示或透传，不需要自己推断。
