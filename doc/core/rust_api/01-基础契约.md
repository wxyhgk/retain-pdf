# 基础契约

这篇只讲最基础的读法。

## 响应包装

所有业务接口统一是：

```json
{
  "code": 0,
  "message": "ok",
  "data": {}
}
```

读取顺序：

1. 先看 HTTP 是否 `2xx`
2. 再看 `code`
3. `code === 0` 再读 `data`

不要直接读顶层 `items`、`stage` 或 `failure`。

## 列表接口

列表页主接口：

- `GET /api/v1/jobs`
- `GET /api/v1/ocr/jobs`

常用字段：

- `job_id`
- `display_name`
- `workflow`
- `status`
- `stage`
- `created_at`
- `updated_at`
- `detail_url`
- `invocation`

推荐只把它当“最近任务入口”，不要在列表里硬塞详情页逻辑。

## Provider 校验

MinerU Token 校验：

- `POST /api/v1/providers/mineru/validate-token`

用途：

- 提交 OCR 前先校验 token 是否可用
- 提前发现过期、无效或网络问题

推荐返回状态：

- `valid`
- `unauthorized`
- `expired`
- `network_error`
- `provider_error`

## 基础读法

- 任务状态看 `status`
- 当前阶段看 `stage`
- 是否能点按钮看 `actions`
- 是否能下载看 `artifacts` / `artifacts-manifest`
- 任务是否真完成看 `status === succeeded`
