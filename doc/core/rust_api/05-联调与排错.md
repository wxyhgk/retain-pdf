# 联调与排错

## 1. 最常见的三类误判

### 1.1 后端没返回，还是前端没读对

先直接用 curl 看原始响应，不要先猜。

例如：

```bash
curl -s http://127.0.0.1:41000/api/v1/jobs/<job_id> \
  -H 'X-API-Key: your-key'
```

以及：

```bash
curl -s 'http://127.0.0.1:41000/api/v1/jobs/<job_id>/events?limit=50&offset=0' \
  -H 'X-API-Key: your-key'
```

如果原始响应有值，而页面没有，优先查前端解包路径。

### 1.2 “后端未返回 runtime.stage_history”

这句话有两种完全不同的情况：

第一种，当前任务是历史老任务：

- `runtime = null`
- 说明当时后端还未持久化运行态时间线

第二种，当前任务是新任务但后端真的没写：

- 这才是需要排查的异常

不要把这两种情况混成一个问题。

### 1.3 “详情或下载怎么直接报旧任务不支持”

这是另一类问题，不要和 `runtime = null` 混淆。

如果任务仍使用：

- `originPDF/jsonPDF/transPDF/typstPDF` 旧目录布局
- 旧版绝对路径 artifact 存储

那么当前后端会直接拒绝详情/下载路径，要求重跑。

这不是暂时性故障，也不是前端解析问题，而是当前主链已经停止兼容旧任务布局。

### 1.4 “事件流为空”

优先检查：

1. 接口是不是 `200`
2. `data.items` 是否非空
3. 前端是不是错误读取为顶层 `items`

## 2. 任务详情页的推荐取值表

- 当前状态：`data.status`
- 当前阶段：`data.runtime.current_stage`，兜底 `data.stage`
- 当前阶段说明：`data.stage_detail`
- 过程时间线：`data.runtime.stage_history`
- 当前阶段耗时：`data.runtime.active_stage_elapsed_ms`
- 总耗时：`data.runtime.total_elapsed_ms`
- 失败摘要：`data.failure.summary`
- 失败类别：`data.failure.category`
- 公式翻译模式：`data.request_payload.translation.math_mode`
- 执行协议：`data.invocation`
- 归一化摘要：`data.normalization_summary`
- 术语表摘要：`data.glossary_summary`
- 下载按钮：`data.actions.*.enabled`

## 3. 事件流 tab 的推荐取值表

- 事件数组：`data.items`
- 分页 limit：`data.limit`
- 分页 offset：`data.offset`

每个 item 重点展示：

- `seq`
- `ts`
- `level`
- `stage`
- `event`
- `message`

`payload` 建议作为展开区，不要默认全部摊开。

## 4. 磁盘上怎么查

任务根目录一般在：

`DATA_ROOT/jobs/{job_id}/`

常见排查位置：

- `specs/`
- `logs/events.jsonl`
- `artifacts/pipeline_summary.json`
- `ocr/`
- `translated/`
- `rendered/`

如果任务失败且需要看更细的诊断，新增重点位置：

- `logs/failure-ai-diagnosis.request.json`
- `logs/failure-ai-diagnosis.response.json`

说明：

- 这两个文件只会在主失败类别仍为 `unknown` 且后端成功触发 AI 补充诊断时出现
- 没有这两个文件，不代表失败分类坏了，很多任务本来就能直接通过规则分类
- `specs/` 目录里的 stage spec 是当前 worker 真正执行的输入协议；如果这里不存在，但你以为是新任务，先排查任务是不是旧产物或半成品目录

## 5. 一个真实案例

任务 `20260404150516-75857c`：

- 详情接口里 `runtime = null`
- 所以“过程时间线”不能显示
- 但 `/events` 接口是有数据的
- 磁盘上也有 `logs/events.jsonl`

这说明：

- 该任务属于运行态时间线持久化上线前的历史任务
- 不是事件流坏了
- 也不是前端时间线 tab 自己渲染失败

## 6. 文档级结论

前端如果只需要稳定展示：

- 时间线只看 `runtime.stage_history`
- 事件流只看 `/events`
- 执行协议只看 `invocation`
- 归一化概览只看 `normalization_summary`
- 不要互相回填，不要混用，不要从事件流反推主时间线

## 7. 失败归因现在怎么读

任务失败后，推荐按这个顺序看：

1. `data.failure.summary`
2. `data.failure.root_cause`
3. `data.failure.suggestion`
4. `data.failure.raw_diagnostic`
5. `data.failure.ai_diagnostic`

判断原则：

- `failure` 是主真源
- `failure.raw_diagnostic` 用来回答“原始异常到底是什么”
- `failure.ai_diagnostic` 用来回答“如果规则没识别出来，AI 认为大概率是什么”
- `failure_diagnostic` 只是兼容旧字段，不要再把它当主真源

当前后端失败诊断链路：

1. Python 顶层入口先输出结构化失败 JSON
2. Rust 优先解析这份结构化失败并分类
3. 如果还是 `unknown`，再尝试追加 AI 补充诊断
4. 最终结果持久化到任务详情和事件流

## 8. 事件流里和失败诊断相关的重点事件

常看这几个：

- `failure_classified`
- `failure_ai_diagnosed`
- `job_terminal`

含义：

- `failure_classified`：后端已经得出结构化失败分类
- `failure_ai_diagnosed`：仅 `unknown` 失败才可能出现，表示 AI 已补充诊断
- `job_terminal`：任务进入最终终态，适合读取最终摘要

## 9. OCR 页码范围联调要点

`POST /api/v1/jobs` 里如果要限制 OCR 页码范围，字段位置是：

```json
{
  "ocr": {
    "page_ranges": "1-5"
  }
}
```

当前后端行为：

- `ocr.page_ranges` 会进入任务持久化的 `request_payload`
- `provider=mineru` 时：
  - `source.upload_id` 上传 PDF 路径会先裁剪出子集 PDF，再上传该子集
  - `source.source_url` 远程 URL 路径也会透传
- 空字符串表示不限制页码

如果 `source.upload_id + ocr.page_ranges` 已经生效，那么任务目录里的 `source_pdf`、后续翻译页码和最终渲染 PDF 都应该只覆盖这个子集，不应该再出现“整本 PDF 里只有部分页被翻译”的结果。

如果怀疑页码范围没生效，先查这三层，不要直接猜前端：

1. 请求详情里 `request_payload.ocr.page_ranges` 是否非空
2. 数据库 `jobs.request_json` 中 `$.ocr.page_ranges` 是否非空
3. 再看 provider 执行日志，而不是先怀疑页面表单

## 11. `math_mode` 联调要点

如果你在排查“公式块为什么走得慢”或“为什么没有 placeholder”：

1. 先看 `request_payload.translation.math_mode`
2. `placeholder` 表示当前任务仍在走旧公式保护链
3. `direct_typst` 表示当前任务走实验性的公式直出分支

判断原则：

- `placeholder` 下出现 placeholder 校验、公式分段、windowed formula，都属于预期
- `direct_typst` 下不应再把 placeholder 稳定性当主问题，重点应看模型直出质量、英文残留和渲染兼容

## 10. 旧任务被拒绝时怎么排

如果详情或下载接口返回的是“旧任务不支持，需要重跑”，按下面顺序确认：

1. 看任务目录是否还是 `originPDF/jsonPDF/transPDF/typstPDF`
2. 看数据库 artifacts 存储是否仍是绝对路径
3. 看该任务是否早于本轮 spec-only 与新 artifact 布局切换

结论：

- 这种任务当前不会自动迁移
- 不建议再写临时兼容代码兜它
- 正确处理方式就是重跑
