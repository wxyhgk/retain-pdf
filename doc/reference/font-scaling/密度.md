# 密度到底是怎么判断的？

如果把一个文字框想象成一个纸盒子，那么“密度”其实就在回答一句话：

**这段翻译后的内容，按当前字号和行距，塞进这个盒子里，到底是宽松、刚好，还是已经过载？**

这个问题看起来很直觉，但真要落成代码，通常不会只看一个指标。因为“挤不挤”至少有两层含义：

- 内容本身是不是变长了
- 这个框本身有没有能力容纳这些内容

所以在当前实现里，密度并不是一个单独的常量，而是由几组函数一起判断出来的。

---

## 一、先说结论：我们实际上在看两种密度

当前实现里，跟“密度”最相关的其实不是一个函数，而是两条线：

1. **长度密度**
   - 看翻译后内容相对原文是不是“膨胀”了
   - 对应函数：`translation_density_ratio(...)`

2. **布局密度**
   - 看按当前字号和行距，内容在这个框里会不会显得太挤
   - 对应函数：`layout_density_ratio(...)`

在 [fit.py](../../backend/scripts/services/rendering/layout/payload/fit.py) 里，这两个指标会一起参与判断：

```python
length_density_ratio = translation_density_ratio(item, protected_text)
layout_density = layout_density_ratio(box, protected_text, font_size_pt=font_size_pt, line_step_pt=line_step)
is_dense_block = length_density_ratio >= COMPACT_TRIGGER_RATIO or layout_density >= LAYOUT_COMPACT_TRIGGER_RATIO
```

也就是说，系统不是只问“翻译是不是变长了”，也不是只问“框是不是快满了”，而是两个都问。

---

## 二、第一层：内容有没有明显膨胀

最直接的思路是：原文只有一点点词，翻译后却变成很长一段，那它大概率就更难排。

这一层由 [text_common.py](../../backend/scripts/services/rendering/layout/payload/text_common.py) 里的 `translation_density_ratio(...)` 负责：

```python
def translation_density_ratio(item: dict, protected_text: str) -> float:
    source_words = source_word_count(item)
    if source_words <= 0:
        return 0.0
    zh_chars = translated_zh_char_count(protected_text)
    if zh_chars <= 0:
        return 0.0
    return zh_chars / source_words
```

这个函数做的事情非常简单：

- 先数原文大概有多少个英文词：`source_word_count(item)`
- 再数翻译后有多少个中文字符：`translated_zh_char_count(protected_text)`
- 用“中文字符数 / 原文词数”得到一个比例

它的意义不是“精确计算排版”，而是快速判断：

**这段翻译后，是不是比原文在视觉上更容易变得拥挤。**

### 例子

假设一个原文块：

- 原文词数：20
- 翻译后中文字符：18

那么：

`translation_density_ratio = 18 / 20 = 0.9`

这说明它已经到了一个偏紧的边缘。

如果另一个块：

- 原文词数：20
- 翻译后中文字符：24

那么：

`translation_density_ratio = 24 / 20 = 1.2`

这种块就属于“明显膨胀”，后面通常要更保守地对待。

当前阈值也写在同一个文件里：

- `COMPACT_TRIGGER_RATIO = 0.9`
- `HEAVY_COMPACT_RATIO = 1.0`

直白地说：

- `>= 0.9`：说明开始变紧了
- `>= 1.0`：说明已经是重度紧凑块

## 三、第二层：按当前字号，这个框会不会真的被塞满

上一层只能说明“内容变长了没有”，但它没有看框。

同样是 `ratio = 1.0` 的两段话：

- 放在一个宽 400pt 的正文框里，可能完全没问题
- 放在一个宽 160pt 的图注框里，可能马上爆掉

所以第二层必须看“框的容量”。

这一步由 [text_common.py](../../backend/scripts/services/rendering/layout/payload/text_common.py) 里的 `layout_density_ratio(...)` 负责：

```python
def layout_density_ratio(
    inner: list[float],
    protected_text: str,
    *,
    font_size_pt: float,
    line_step_pt: float,
) -> float:
    width = max(8.0, inner[2] - inner[0])
    height = max(8.0, inner[3] - inner[1])
    zh_chars = translated_zh_char_count(protected_text)
    approx_char_width = max(font_size_pt * 0.92, 1.0)
    chars_per_line = max(4.0, width / approx_char_width)
    required_lines = max(1.0, zh_chars / chars_per_line)
    occupied_height = required_lines * line_step_pt
    return occupied_height / height
```

这个函数的逻辑可以翻译成人话：

1. 先看框有多宽、多高
2. 假设当前字号下，一个字大概有多宽
3. 推出一行大概能放多少字
4. 再估计这段翻译后文本大概要占几行
5. 最后算出这些行会占掉多少高度
6. 再拿占用高度去除以框高度

得到的结果就是一个非常直观的比值：

- `< 1.0`：理论上还放得下
- `≈ 1.0`：已经非常紧
- `> 1.0`：理论上已经超框

### 例子

假设一个框：

- 宽度：180pt
- 高度：90pt
- 当前字号：9pt
- 当前行距：12pt
- 翻译后中文字符：72

粗略估算：

- 单字宽度约为 `9 × 0.92 = 8.28pt`
- 每行大概可放 `180 / 8.28 ≈ 21.7` 个字
- 72 个字大概需要 `72 / 21.7 ≈ 3.3` 行
- 占用高度约 `3.3 × 12 = 39.6pt`
- 布局密度约 `39.6 / 90 = 0.44`

这说明这个块其实不挤。

如果同样内容放到另一个框：

- 宽度：110pt
- 高度：48pt

那么：

- 每行大概只能放 `110 / 8.28 ≈ 13.3` 个字
- 72 个字需要 `72 / 13.3 ≈ 5.4` 行
- 占用高度约 `5.4 × 12 = 64.8pt`
- 布局密度约 `64.8 / 48 = 1.35`

这就是典型的高密度块，当前字号一定太大。

## 四、第三层：框的“真实容量”到底怎么算

上面那个 `layout_density_ratio(...)` 是一个快速估算，它偏轻量，适合先做密度判断。

真正更接近“这个框到底能装多少内容”的计算，在 [capacity.py](../../backend/scripts/services/rendering/layout/payload/capacity.py)。

最核心的是 `box_capacity_units(...)`：

```python
def box_capacity_units(
    inner: list[float],
    font_size_pt: float,
    leading_em: float,
    visual_lines: int | None = None,
) -> float:
    width = max(8.0, inner[2] - inner[0])
    height = max(8.0, inner[3] - inner[1])
    line_step = max(font_size_pt * 1.02, font_size_pt * (1.0 + leading_em))
    lines = max(1, int(height / line_step))
    if visual_lines and visual_lines > 1:
        lines = min(lines, max(1, visual_lines + 1))
    chars_per_line = max(4.0, width / max(font_size_pt * 0.92, 1.0))
    return lines * chars_per_line * 0.98
```

它在做三件事：

1. 按字号和行距推算一共能放多少行
2. 按框宽推算一行能放多少内容
3. 两者相乘，得到这个框的总容量

这里有一个很关键的细节：

`visual_lines`

也就是说，它不是完全相信“框高能塞多少行”，而是会参考 OCR / 版面结构对原块的视觉行数估计，避免把容量想得过于乐观。

---

## 五、第四层：内容需求不是单纯数字符

如果容量算出来了，那“需求”怎么计算？

这由同一个文件里的 `text_demand_units(...)` 负责：

```python
def text_demand_units(protected_text: str, formula_map: list[dict]) -> float:
    formula_lookup = {entry["placeholder"]: entry["formula_text"] for entry in formula_map}
    return sum(token_units(token, formula_lookup) for token in tokenize_protected_text(protected_text))
```

它的意思是：

- 先把文本拆成 token
- 普通文字算普通单位
- 公式占位符不是按 1 个字符算，而是按更接近真实视觉成本的方式算

这一步很重要，因为如果你只是看字符数，就会低估公式的压力。

### 例子

下面两段文本，字符数可能差不多：

1. `该方法显著提高了材料性能。`
2. `该方法显著提高了[[FORMULA_1]]条件下材料性能。`

但第二段因为带公式，实际排版压力更大。  

所以系统不会把它们当成相同需求，而是通过 `token_units(...)` 给公式更高的视觉成本。

---

## 六、第五层：为什么还要引入 visual line count

有一个很容易忽略的问题：

**有时候 OCR 给你的“行数”不可信。**

比如一个本来已经换成 4 行的段落，被 OCR 粘成 1 行。这时如果你只看原始 `lines`，就会严重高估这个框的可排空间。

所以 [measurement.py](../../backend/scripts/services/rendering/layout/typography/measurement.py) 里专门有一套函数在修正这个事情：

- `plain_text_chars_per_line(...)`
- `_predicted_wrapped_line_count(...)`
- `visual_line_count(...)`
- `is_tall_single_line_glue(...)`

其中 `visual_line_count(...)` 的思想是：

- 先看 OCR 说有几行
- 再根据文本长度、框宽、单行字符承载能力，估计“如果正常换行，应该有几行”
- 如果预测的行数明显高于 OCR 行数，就用更保守的行数

这一步的目的不是为了算字号，而是为了防止密度判断被假数据带偏。

### 一个典型例子

假设某块：

- OCR 只给了 1 行
- 但框很高，文本长度有 140 个字符
- 几何上看，一行根本不可能放下这么多内容

这时 `visual_line_count(...)` 会认为：

“这很可能不是 1 行正文，而是 OCR 把一个多行段落粘成了一行。”

于是系统会用预测值去修正后续的容量判断。这样算出来的密度才更接近真实情况。

## 七、密度最终是怎么影响字号的

前面这些函数并不会直接给出“最终字号”，它们的作用更像是给排版器提供判断依据。

最直接的落点是在 [fit.py](../../backend/scripts/services/rendering/layout/payload/fit.py) 的 `fit_translated_block_metrics(...)`：

```python
capacity = box_capacity_units(box, font_size_pt, leading_em, visual_lines=visual_lines)
if capacity <= 0 or (demand <= capacity * 0.96 and layout_density < LAYOUT_DENSITY_SAFE_MAX):
    return font_size_pt, leading_em
```

这里的逻辑非常关键：

- 如果需求没有逼近容量
- 而且布局密度也没有过高

那当前字号就可以保留。

反过来，如果：

- `demand > capacity`
- 或者 `layout_density` 已经过高

那就进入缩小字号、压紧行距的流程。

也就是说，密度不是直接输出“9.2pt”或者“8.8pt”，而是决定：

- 要不要缩
- 缩多少步
- 是只缩字号，还是连行距一起压

---

## 八、可以把它理解成一个很朴素的判断链

如果把所有函数压缩成一句人话，大概就是下面这个链条：

1. **先看翻译后的内容是不是明显变长了**
   - `translation_density_ratio(...)`

2. **再看按当前字号，这段内容在这个框里会占掉多少高度**
   - `layout_density_ratio(...)`

3. **再更严谨一点，估这个框真正能装多少单位内容**
   - `box_capacity_units(...)`

4. **同时别完全相信 OCR 行数，要用 `visual_line_count(...)` 做修正**

5. **最后用“需求 vs 容量”决定是否缩字号**
   - `text_demand_units(...)` vs `box_capacity_units(...)`

---

## 九、一个完整例子

假设某个翻译块有这些条件：

- 框宽：145pt
- 框高：62pt
- 初始字号：9.4pt
- 行距：0.58em
- 原文词数：18
- 翻译后中文字符：22
- 带 2 个公式

系统会这样看：

### 第一步：内容有没有膨胀

`translation_density_ratio = 22 / 18 ≈ 1.22`

这已经属于重度紧凑块。

### 第二步：当前字号下布局会不会太挤

按 `layout_density_ratio(...)` 估算：

- 这个框较窄
- 9.4pt 下每行能放的字数有限
- 加上公式后，真实换行压力更大

算出来的布局密度可能接近甚至超过 `1.0`

### 第三步：再看容量和需求

- `box_capacity_units(...)` 会发现框的总容量偏小
- `text_demand_units(...)` 会因为公式而把需求抬高

于是形成结论：

**这是一个高密度块，当前字号不安全。**

### 第四步：进入收缩

`fit_translated_block_metrics(...)` 会开始尝试：

- 先把字号每次减一点
- 如果还不够，再压一点 leading
- 一直试到需求不再明显超出容量

这就是“密度判断如何真正影响字号”的完整过程。

---

## 十、最后一句总结

所谓“框的密度”，本质上不是在问：

“这个框里有多少字？”

而是在问：

**在当前字号和行距下，这个框的可用容量，和这段翻译内容的真实需求之间，差距还有多少。**

当前实现里，这个问题是由下面几组函数联合回答的：

- 页面与原始行信息修正：
  - [measurement.py](../../backend/scripts/services/rendering/layout/typography/measurement.py)
- 文本长度与布局密度估算：
  - [text_common.py](../../backend/scripts/services/rendering/layout/payload/text_common.py)
- 容量与需求计算：
  - [capacity.py](../../backend/scripts/services/rendering/layout/payload/capacity.py)
- 最终是否收缩字号：
  - [fit.py](../../backend/scripts/services/rendering/layout/payload/fit.py)

如果你继续往前追，“字号为什么最后变小了”，通常答案都可以回到这句话：

**因为当前块的内容需求，已经逼近或超过了当前框在这个字号下的容量。**
