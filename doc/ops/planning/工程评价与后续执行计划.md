# 工程评价与后续执行计划

这篇文档是基于外部评价、当前仓库现状，以及最近一轮实际开发与交付问题整理出的内部判断。

目标不是重复 README，而是回答三个问题：

1. 这个项目目前做得怎么样
2. 当前真正的短板是什么
3. 接下来应该按什么顺序继续做

## 1. 当前项目处于什么阶段

当前项目已经不是单纯的脚本集合，也不是只能演示的 demo。

它已经具备下面这些特征：

- 有浏览器前端
- 有 Rust API 服务层
- 有 Python 处理管线
- 有 Docker 交付形态
- 有桌面端打包链路
- 有任务目录、事件流、产物下载、阶段诊断这些产品化能力

更准确的说法是：

- 它已经是一个“能工作的产品原型”
- 但还没有完全进入“低维护成本、稳定交付”的工程状态

## 2. 外部评价中哪些判断是对的

### 2.1 README 的导航思路是对的

现在仓库确实已经不是一个文件能讲清楚的项目。

用 README 做总导航，而不是把所有细节都堆进去，这是正确方向。

### 2.2 前后端加管线的整体形态是完整的

当前仓库已经覆盖：

- 前端交互
- 任务提交与查询
- OCR 接入
- 翻译处理
- 排版重建
- 产物下载
- Docker 和桌面端交付

从产品链路上说，这个项目是完整的。

### 2.3 Rust API + Python pipeline 的组合是合理的

这个架构并不奇怪，反而很适合当前场景：

- Rust 负责服务端、任务调度、状态管理、接口与交付
- Python 负责 OCR、翻译、排版、模型生态、快速实验

这条技术路线本身没有问题。

## 3. 外部评价没有完全点透，但当前更关键的问题

外部评价提到了“文档一致性、边界划分、可复现构建”，这些都对。

但从最近实际踩坑看，真正更核心的问题是：

## 3.1 系统还缺少足够硬的“单一真源”

很多问题表面看是：

- GitHub Actions 报错
- 桌面端构建失败
- 前端拿不到字段
- 下载路径变化后接口要改

本质上其实是下面这些真源还不够统一：

- 文档真源
- 配置真源
- 产物真源
- 桌面构建输入真源
- 路径与目录结构真源

也就是：

- 本地有的文件，CI 不一定有
- 某个参数本地默认可用，干净环境不一定有
- 某个下载路径是代码里推导的，不是数据库登记的
- 某个字段文档里有，但接口并不稳定返回

这类问题不解决，后面每加一个功能，维护成本都会放大。

## 3.2 目前最大短板不是算法，而是工程一致性

项目的核心卖点当然是：

- 保留排版
- 翻译质量
- PDF 最终效果

但当前最消耗时间的问题，并不主要来自模型能力不足，而是来自：

- 长块翻译不稳定
- 公式占位符保护不足
- 块级失败后的降级策略不够稳
- 阶段诊断不够直接
- 构建链路在本地和 CI 之间不一致
- 文档和代码结构偶尔脱节

换句话说：

- 现在最缺的不是“再接一个更强模型”
- 而是“把现有链路做得更稳、更可解释、更容易定位问题”

## 3.3 仓库大不是原罪，边界不清才是问题

当前仓库是一个单仓多组件工程，这本身是可以接受的。

真正的问题不是目录多，而是这些目录之间的契约还不够硬，比如：

- 前端依赖哪些稳定接口
- 后端哪些字段是长期承诺
- Python 只应该接收哪些参数
- 下载功能到底依赖目录推导还是数据库登记
- 桌面打包到底依赖哪些显式文件

只要边界清楚，单仓完全可以维护。

## 4. 当前阶段应该怎么判断项目是否“够成熟”

如果要判断这个项目是否真的成熟，不要只看“能不能跑出 PDF”，更应该看下面四件事。

### 4.1 干净环境能否稳定构建

判断标准：

- 换一台机器或者 GitHub Actions
- 不依赖本地历史文件
- 不依赖手工补资源
- 能直接产出可运行包

最近 GitHub Actions 连续报错，正说明这部分还在补课阶段。

### 4.2 接口契约是否稳定

判断标准：

- 文档里写了的字段，接口就稳定返回
- 前端展示时间线，不需要再猜事件流如何拼装
- 产物下载路径由后端统一暴露，不需要前端猜目录

### 4.3 任务与产物是否可追踪

判断标准：

- 每个任务发生了什么可以查
- 每个阶段耗时可以查
- 为什么失败可以定位
- 最终应该下载什么文件可以稳定获取

### 4.4 翻译链路是否具备稳定性机制

判断标准：

- 长段落不会轻易炸掉
- 公式/占位符不容易丢
- 出现异常时能自动降级
- 用户看到的是可解释的失败原因，而不是一句“任务失败”

## 5. 当前项目最值得继续投入的方向

如果从“产品效果”和“工程回报”一起看，后续最值得投入的方向，不是盲目扩功能，而是下面三条主线。

### 5.1 第一主线：交付工程收口

目标：

- GitHub Actions 稳定打包
- Windows 桌面端可复现构建
- Docker、前端、后端的交付材料明确
- 不再依赖本地隐式文件

这是把项目从“我机器上能跑”推进到“别人机器上也能稳定交付”的前提。

### 5.2 第二主线：后端契约继续做硬

目标：

- API 文档和真实返回保持一致
- `runtime.stage_history`、`events`、`artifacts` 这些结构长期稳定
- 下载能力走数据库登记，而不是靠猜目录
- 请求参数、任务持久化、下游透传链路可追踪

这是降低前后端联调成本的关键。

### 5.3 第三主线：翻译稳定性优先于“更花哨的增强”

目标：

- 长文块切分更稳
- 公式与占位符保护更稳
- 失败重试与降级策略更稳
- 页面级质量诊断更直观

这里比 RAG、术语库增强更优先。

原因很简单：

- 如果基础稳定性不够，接再多增强能力都会被不稳定结果抵消
- 只有基础链路稳定了，术语表、缩写表、RAG 才会真正放大价值

## 6. 后续三阶段建议

## 阶段一：把工程真源统一起来

优先做：

- 统一文档入口
- 统一版本真源
- 统一桌面打包输入
- 统一配置来源
- 统一产物登记方式

验收标准：

- 干净环境不会再因为缺少某个本地文件而构建失败
- 文档不会让人分不清哪个是当前有效入口

## 阶段二：把 API 与任务状态层做成稳定契约

优先做：

- 固化任务详情结构
- 固化阶段时间线结构
- 固化事件流结构
- 固化产物清单结构
- 固化错误诊断结构

验收标准：

- 前端不需要猜字段
- 前端不需要自己重建后端本应直接给出的状态

## 阶段三：集中打磨翻译质量稳定性

优先做：

- 长块切分策略
- 公式/占位符保护
- 单块失败回退
- 页面级质量诊断
- 质量问题的可观测性

验收标准：

- 同一类 PDF 的成功率明显提高
- 用户看到的“失败”变少
- 即使失败，也能知道到底坏在什么阶段、什么块

## 7. 一句话结论

这个项目目前最强的地方，不是某个单点算法，而是已经把“PDF 上传 -> OCR -> 翻译 -> 排版重建 -> 下载交付”整条链路真正做起来了。

当前最需要继续投入的，不是盲目加更多能力，而是把这条链路的：

- 工程一致性
- 契约稳定性
- 构建可复现性
- 翻译稳定性

继续打磨到可长期维护的水平。

如果这四件事做扎实，这个项目就会从“很强的原型”真正进入“可稳定交付的产品工程”阶段。
